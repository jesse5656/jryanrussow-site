# Systems Architect Design Doctrine

Version: 0.1.0

Status:
Living Doctrine

------------------------------------------------------------------------------

Purpose

The Design Doctrine defines how Systems Architects approach problems.

It is not a collection of techniques.

It is a way of thinking.

------------------------------------------------------------------------------

Doctrine I

Seek understanding before intervention.

Do not improve what you do not yet understand.

------------------------------------------------------------------------------

Doctrine II

Treat symptoms as evidence.

Treat systems as causes.

------------------------------------------------------------------------------

Doctrine III

Preserve what works.

Improve what doesn't.

Do not redesign simply because something is old.

------------------------------------------------------------------------------

Doctrine IV

Every decision has downstream consequences.

Think in generations, not transactions.

------------------------------------------------------------------------------

Doctrine V

Document before memory fades.

Uncaptured knowledge is eventually lost.

------------------------------------------------------------------------------

Doctrine VI

Design systems that create better stewards,
not greater dependence.

------------------------------------------------------------------------------

Doctrine VII

Assume every system will eventually outlive its architect.

Design accordingly.

------------------------------------------------------------------------------

Doctrine VIII

Technology exists to amplify human judgment,
not replace it.

------------------------------------------------------------------------------

Doctrine IX

Complexity should exist only where reality requires it.

------------------------------------------------------------------------------

Doctrine X

Truth outranks ego.

Evidence outranks preference.

Stewardship outranks ownership.
